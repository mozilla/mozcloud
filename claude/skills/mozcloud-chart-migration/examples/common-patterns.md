## Deployment → Workload

### Pattern: Basic Deployment

**Custom Template** (`templates/deployment.yaml`):
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
  labels:
    app: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app
        image: registry/my-app:v1.0.0
        ports:
        - containerPort: 8080
        env:
        - name: ENV_VAR
          value: "value"
        resources:
          limits:
            cpu: 500m
            memory: 512Mi
          requests:
            cpu: 100m
            memory: 256Mi
```

**Mozcloud Configuration** (`values.yaml`):
```yaml
workloads:
  my-app:  # Key must match deployment name to preserve resource name
    type: deployment
    replicas: 3
    image: registry/my-app:v1.0.0
    ports:
      - name: http
        containerPort: 8080
    env:
      ENV_VAR: "value"
    resources:
      limits:
        cpu: 500m
        memory: 512Mi
      requests:
        cpu: 100m
        memory: 256Mi
```

**Key Points**:
- Workload key (`my-app`) becomes the Deployment name
- Environment variables use map format instead of array
- Ports are optional if using standard HTTP/8080
- Labels are auto-generated by mozcloud

### Pattern: Deployment with initContainers

**Custom Template**:
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  template:
    spec:
      initContainers:
      - name: migration
        image: registry/my-app:v1.0.0
        command: ['python', 'manage.py', 'migrate']
      containers:
      - name: my-app
        image: registry/my-app:v1.0.0
```

**Mozcloud Configuration**:
```yaml
workloads:
  my-app:
    type: deployment
    image: registry/my-app:v1.0.0

    initContainers:
      - name: migration
        image: registry/my-app:v1.0.0
        command: ['python', 'manage.py', 'migrate']
```

### Pattern: Deployment with Volume Mounts

**Custom Template**:
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  template:
    spec:
      containers:
      - name: my-app
        image: registry/my-app:v1.0.0
        volumeMounts:
        - name: config
          mountPath: /etc/config
        - name: data
          mountPath: /data
      volumes:
      - name: config
        configMap:
          name: my-app-config
      - name: data
        persistentVolumeClaim:
          claimName: my-app-data
```

**Mozcloud Configuration**:
```yaml
workloads:
  my-app:
    type: deployment
    image: registry/my-app:v1.0.0

    volumeMounts:
      - name: config
        mountPath: /etc/config
      - name: data
        mountPath: /data

    volumes:
      - name: config
        configMap:
          name: my-app-config
      - name: data
        persistentVolumeClaim:
          claimName: my-app-data
```

### Pattern: Deployment with Secrets

**Custom Template**:
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  template:
    spec:
      containers:
      - name: my-app
        image: registry/my-app:v1.0.0
        env:
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: my-app-secrets
              key: db-password
        - name: API_KEY
          valueFrom:
            secretKeyRef:
              name: my-app-secrets
              key: api-key
```

**Mozcloud Configuration**:
```yaml
workloads:
  my-app:
    type: deployment
    image: registry/my-app:v1.0.0

    secrets:
      - secretName: my-app-secrets
        keys:
          - DB_PASSWORD
          - API_KEY
```

**Key Points**:
- Mozcloud auto-maps secret keys to environment variables
- Secret key names must match environment variable names
- If secret key differs from env var, use `envVarName: secretKeyName` mapping

## Service

### Pattern: ClusterIP Service

**Custom Template** (`templates/service.yaml`):
```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  type: ClusterIP
  ports:
  - port: 80
    targetPort: 8080
    protocol: TCP
    name: http
  selector:
    app: my-app
```

**Mozcloud Configuration**:
```yaml
workloads:
  my-app:
    type: deployment
    image: registry/my-app:v1.0.0

    service:
      enabled: true
      type: ClusterIP
      ports:
        - name: http
          port: 80
          targetPort: 8080
```

**Key Points**:
- Service is nested under workload configuration
- Service name matches workload name (my-app)
- Selector is auto-generated to match workload pods

---

### Pattern: Service with Multiple Ports

**Custom Template**:
```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  type: ClusterIP
  ports:
  - port: 80
    targetPort: 8080
    name: http
  - port: 9090
    targetPort: 9090
    name: metrics
```

**Mozcloud Configuration**:
```yaml
workloads:
  my-app:
    type: deployment
    image: registry/my-app:v1.0.0

    ports:
      - name: http
        containerPort: 8080
      - name: metrics
        containerPort: 9090

    service:
      enabled: true
      ports:
        - name: http
          port: 80
          targetPort: 8080
        - name: metrics
          port: 9090
          targetPort: 9090
```

## ConfigMap

### Pattern: Inline Data ConfigMap

**Custom Template** (`templates/configmap.yaml`):
```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: my-app-config
data:
  app.conf: |
    server {
      listen 8080;
      server_name localhost;
    }
  settings.json: |
    {
      "debug": false,
      "timeout": 30
    }
```

**Mozcloud Configuration**:
```yaml
configs:
  my-app-config:
    data:
      app.conf: |
        server {
          listen 8080;
          server_name localhost;
        }
      settings.json: |
        {
          "debug": false,
          "timeout": 30
        }
```

**Key Points**:
- ConfigMap name is the key in `configs` map
- Data structure remains identical
- Resource name preserved: `my-app-config`

---

### Pattern: ConfigMap from Files

**Custom Template**:
```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: my-app-config
data:
  {{ (.Files.Glob "configs/*").AsConfig | nindent 2 }}
```

**Mozcloud Configuration**:
```yaml
configs:
  my-app-config:
    files:
      - configs/app.conf
      - configs/settings.json
```

**File Structure**:
```
chart/
├── configs/
│   ├── app.conf
│   └── settings.json
└── values.yaml
```

**Key Points**:
- Use `files` instead of `data` for file-based configs
- Files are read from chart directory
- Supports glob patterns: `configs/*`

## ExternalSecret

### Pattern: GCP Secrets Manager

**Custom Template** (`templates/externalsecret.yaml`):
```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: my-app-secrets
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: gcpsm-secret-store
    kind: SecretStore
  target:
    name: my-app-secrets
    creationPolicy: Owner
  data:
  - secretKey: database-url
    remoteRef:
      key: my-app-db-credentials
      property: url
  - secretKey: api-key
    remoteRef:
      key: my-app-api-key
```

**Mozcloud Configuration**:
```yaml
externalSecrets:
  my-app-secrets:
    backendType: gcpSecretsManager
    projectId: my-gcp-project
    data:
      - key: my-app-db-credentials
        name: database-url
        property: url
      - key: my-app-api-key
        name: api-key
```

**Key Points**:
- ExternalSecret name is key in `externalSecrets` map
- `name` field is the secret key (target in Kubernetes Secret)
- `key` is the GCP Secret Manager secret name
- `property` is optional (for JSON secrets)
- Target Secret name matches ExternalSecret name

---

## Ingress

Mozcloud supports Ingress resources natively - no conversion to HTTPRoute needed! The configuration is nested under the workload's `hosts` section.

### Pattern: GCP Ingress with Static IP and ManagedCertificate

**Custom Template** (`templates/ingress.yaml`):
```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: cicd-demos
  annotations:
    kubernetes.io/ingress.global-static-ip-name: cicd-demos-dev-ip-v4
    networking.gke.io/managed-certificates: managed-certificate
    kubernetes.io/ingress.class: "gce"
    networking.gke.io/v1beta1.FrontendConfig: cicd-demos
spec:
  defaultBackend:
    service:
      name: cicd-demos
      port:
        number: 8080
```

**Mozcloud Configuration**:
```yaml
workloads:
  cicd-demos:
    type: deployment
    image: us-docker.pkg.dev/project/repo/app:tag

    service:
      enabled: true
      ports:
        - name: http
          port: 8080
          targetPort: 8080

    hosts:
      cicd-demos:
        api: ingress  # Creates an Ingress resource
        domains:
          - dev.cicd-demos.nonprod.sandbox.mozgcp.net
        addresses:
          - cicd-demos-dev-ip-v4  # Static IP name
        tls:
          type: ManagedCertificate  # GCP ManagedCertificate

# FrontendConfig (GCP-specific, if needed)
frontendConfigs:
  cicd-demos:
    sslPolicy: "mozilla-intermediate"
```

**Key Points**:
- Ingress is created **natively** by mozcloud (no HTTPRoute conversion)
- Configuration is nested under `workloads.<name>.hosts`
- `api: ingress` specifies to create an Ingress resource
- `domains` specifies the hostnames
- `addresses` specifies GCP static IP address names
- `tls.type: ManagedCertificate` uses GCP's managed certificates
- FrontendConfig/BackendConfig supported for GCP-specific settings

---

### Pattern: Simple Ingress with cert-manager

**Custom Template** (`templates/ingress.yaml`):
```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-app
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  ingressClassName: nginx
  tls:
  - hosts:
    - my-app.example.com
    secretName: my-app-tls
  rules:
  - host: my-app.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: my-app
            port:
              number: 80
```

**Mozcloud Configuration**:
```yaml
workloads:
  my-app:
    type: deployment
    image: registry/my-app:v1.0.0

    service:
      enabled: true
      ports:
        - name: http
          port: 80
          targetPort: 8080

    hosts:
      my-app:
        api: ingress
        ingressClassName: nginx
        domains:
          - my-app.example.com
        tls:
          type: Certificate  # Use cert-manager
          issuerRef:
            name: letsencrypt-prod
            kind: ClusterIssuer
```

**Key Points**:
- For cert-manager TLS: use `tls.type: Certificate`
- For GCP managed certificates: use `tls.type: ManagedCertificate`
- `ingressClassName` can be specified if not using default
- Annotations can be added via `hosts.<name>.annotations`

## Certificate

### Pattern: cert-manager Certificate

**Custom Template** (`templates/certificate.yaml`):
```yaml
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: my-app-tls
spec:
  secretName: my-app-tls
  issuerRef:
    name: letsencrypt-prod
    kind: ClusterIssuer
  dnsNames:
  - my-app.example.com
  - www.my-app.example.com
```

**Mozcloud Configuration**:
```yaml
certificates:
  my-app-tls:
    dnsNames:
      - my-app.example.com
      - www.my-app.example.com
    issuerRef:
      name: letsencrypt-prod
      kind: ClusterIssuer
```

**Key Points**:
- Certificate name is key in `certificates` map
- `secretName` auto-generated (matches certificate name)
- Supports multiple DNS names
- Issuer reference remains the same

## PodDisruptionBudget

### Pattern: PDB for High Availability

**Custom Template** (`templates/pdb.yaml`):
```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: my-app-pdb
spec:
  minAvailable: 1
  selector:
    matchLabels:
      app: my-app
```

**Mozcloud Configuration**:
```yaml
workloads:
  my-app:
    type: deployment
    replicas: 3
    image: registry/my-app:v1.0.0

    podDisruptionBudget:
      enabled: true
      minAvailable: 1
```

**Key Points**:
- PDB nested under workload configuration
- Selector auto-generated to match workload
- PDB name follows pattern: `<workload-name>-pdb`
- Can use `maxUnavailable` instead of `minAvailable`

## NetworkPolicy

### Pattern: Restrict Ingress Traffic

**Custom Template** (`templates/networkpolicy.yaml`):
```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: my-app-netpol
spec:
  podSelector:
    matchLabels:
      app: my-app
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
```

**Mozcloud Configuration**:
```yaml
workloads:
  my-app:
    type: deployment
    image: registry/my-app:v1.0.0

    networkPolicy:
      enabled: true
      policyTypes:
        - Ingress
      ingress:
        - from:
          - podSelector:
              matchLabels:
                app: frontend
          ports:
          - protocol: TCP
            port: 8080
```

**Key Points**:
- NetworkPolicy nested under workload
- PodSelector auto-generated to match workload
- Name follows pattern: `<workload-name>-netpol`

## CronJob

### Pattern: Scheduled Job

**Custom Template** (`templates/cronjob.yaml`):
```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: my-backup-job
spec:
  schedule: "0 2 * * *"
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: backup
            image: registry/backup:v1.0.0
            command: ['./backup.sh']
          restartPolicy: OnFailure
```

**Mozcloud Configuration**:
```yaml
workloads:
  my-backup-job:
    type: cronjob
    schedule: "0 2 * * *"
    image: registry/backup:v1.0.0
    command: ['./backup.sh']
    restartPolicy: OnFailure
```

**Key Points**:
- Use `type: cronjob` instead of `type: deployment`
- Add `schedule` field with cron expression
- `restartPolicy` defaults to `OnFailure` for cronjobs
- Job name matches workload key

## Resource Name Preservation Checklist

When migrating, always verify resource names are preserved:

| Resource Type | Name Source | How to Preserve |
|---------------|-------------|-----------------|
| Deployment | Workload key | Use original deployment name as workload key |
| Service | Workload key | Auto-generated from workload name |
| ConfigMap | Config key | Use original configmap name as config key |
| ExternalSecret | ExternalSecret key | Use original externalsecret name as key |
| Secret (target) | ExternalSecret key | Auto-generated, matches externalsecret name |
| Ingress | Workload hosts key | Auto-generated from workload name or hosts key |
| Certificate | Certificate key | Use original certificate name as key |
| ManagedCertificate | Hosts config | Auto-generated from domains (GCP-specific) |
| ServiceAccount | Workload + suffix | Auto-generated: `<workload-name>-sa` |
| PDB | Workload + suffix | Auto-generated: `<workload-name>-pdb` |

## Summary

**Key Migration Principles**:

1. **Workload = Deployment**: The workload key becomes the Deployment name
2. **Nested Configuration**: Services, PDBs, ServiceAccounts, Ingress nest under workloads
3. **Top-Level Resources**: ConfigMaps, ExternalSecrets, Certificates at top level
4. **Name Preservation**: Always use original resource names as keys
5. **Auto-Generation**: Mozcloud auto-generates labels, selectors, and some resource names
6. **Map vs Array**: Environment variables use maps, not arrays
7. **Native Ingress Support**: Mozcloud supports Ingress resources natively via `workloads.<name>.hosts` configuration

**Testing Pattern**:
```bash
# Render custom chart
helm template . -f values.yaml -f values-dev.yaml > custom.yaml

# Render mozcloud chart
helm template . -f values.yaml -f values-dev.yaml > mozcloud.yaml

# Compare
diff custom.yaml mozcloud.yaml

# Or use render-diff
render-diff -f values-dev.yaml -su
```
